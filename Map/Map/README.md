# Map — планирование маршрута покрытия поля

> Техническая документация подсистемы планирования. Общее описание дипломного проекта — в [README](../README.md) в корне репозитория.

Desktop-приложение на **Qt/C++** для построения маршрута покрытия (coverage path planning, CPP) сельскохозяйственного участка по контуру поля с учётом зон исключения и параметров прохода техники. Результат визуализируется на интерактивной карте; маршрут и команды можно экспортировать в JSON.

## Возможности

- Отображение карты (тайловые слои на базе [QGeoView](https://github.com/alexey-lysiuk/QGeoView))
- Задание границы участка: импорт GeoJSON или рисование контура на карте
- Зоны исключения (вырезы) внутри поля
- Параметры построения: шаг прохода, угол рядов, отступ от границы
- Полный 9-стадийный пайплайн `RoutePipeline` с отладочной визуализацией промежуточных этапов
- Упрощённый режим: параллельные ряды и змейка без учёта вырезов и сшивки островов
- Формирование команд движения и экспорт waypoints в JSON

## Структура каталога

```text
Map/
├── main.cpp
├── GeoViewWidget.*     # главный виджет: карта + UI + состояние
```

## Алгоритм RoutePipeline

Основной путь построения маршрута — пайплайн из девяти последовательных стадий (`features/algorithm/pipeline/`):

| № | Стадия | Назначение |
|---|--------|------------|
| 1 | **Prepare** | Нормализация контура и вырезов, inset (отступ), направления рядов |
| 2 | **Sweep** | Параллельные ряды покрытия, несколько проходов по остаточной области |
| 3 | **Filter** | Отсечение коротких сегментов, группировка по рядам |
| 4 | **Graph** | Граф соседства сегментов между соседними рядами |
| 5 | **Merge** | Слияние компонент связности с геометрическими ограничениями |
| 6 | **Routes** | «Змейка» внутри компонент, разбиение на чанки |
| 7 | **Group** | Объединение чанков в острова маршрута |
| 8 | **Stitch** | Сшивка островов в единую полилинию |
| 9 | **Approach** | Подвод к маршруту покрытия и возврат к стартовой или конечной точке |

Все вычисления внутри пайплайна выполняются в **проекции карты** (Web Mercator, EPSG:3857). Геодезические координаты используются при экспорте и формировании команд.

## Требования

- **CMake** ≥ 3.10
- **Qt 6** (Core, Gui, Widgets, Network) — рекомендуется; **Qt 5** поддерживается в `CMakeLists.txt` как fallback
- **Компилятор C++17** (MinGW на Windows или GCC/Clang на Linux)
- **Boost** (только заголовки, Boost.Geometry) — путь задаётся в `CMakeLists.txt` через `BOOST_ROOT`

### Зависимости, включённые в репозиторий

- **QGeoView** — `../lib/`
- **Clipper 6.4.2** — `clipper/`

## Сборка

Команды выполняются из **корня репозитория**.

### Windows (MinGW + Qt 6)

```powershell
# При необходимости укажите путь к Qt в CMakeLists.txt или через переменную окружения:
# $env:CMAKE_PREFIX_PATH = "C:/Qt/6.8.2/mingw_64"

# Укажите путь к Boost (по умолчанию в Map/CMakeLists.txt: C:/local/boost_1_87_0)
cmake -S . -B build -G "MinGW Makefiles" -DCMAKE_PREFIX_PATH="C:/Qt/6.8.2/mingw_64"
cmake --build build
```

Исполняемый файл: `build/Map/Map.exe` (или `build/Map.exe` в зависимости от генератора).

### Linux

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build
./build/Map/Map
```

> **Примечание.** В `Map/CMakeLists.txt` жёстко задан `BOOST_ROOT` (по умолчанию `C:/local/boost_1_87_0`). На другой машине измените эту строку под свой путь к Boost.

## Использование

Типовой сценарий работы в приложении:

1. **Подготовка** — загрузить контур (GeoJSON) или нарисовать границу поля на карте; при необходимости добавить зоны исключения.
2. **Маршрут** — задать шаг прохода, угол рядов, отступ от границы; построить маршрут полным пайплайном (`RoutePipeline`) или включить упрощённый режим (параллельное покрытие без вырезов).
3. **Проверка** — просмотреть результат на карте; при включённом debug-режиме — промежуточные стадии пайплайна.
4. **Экспорт** — построить команды движения (просмотр JSON в панели Info) и при необходимости сохранить waypoints в файл.

Панель управления разделена на блоки: «1) Подготовка», «2) Маршрут», «3) Команды и экспорт».

## Экспорт

Для waypoints нужны начальная позиция робота на карте и построенный маршрут. При сохранении формируется JSON со структурой:

- `origin` — объект с полями `latitude`, `longitude`, `yaw` (курс в радианах)
- `default_reach_radius_m` — радиус достижения waypoint по умолчанию
- `waypoints` — массив точек (`id` — строка, `latitude`, `longitude`, `reach_radius_m`)

Экспортированный файл передаётся в навигационный модуль (`Navigation/`).

## Лицензия

Библиотека QGeoView и связанные компоненты распространяются на условиях лицензии, указанной авторами QGeoView.
